## 1. REST APIとは？

### 概念
**REST API（Representational State Transfer API）**は、「Webを介してシステム同士がスムーズに対話するためのルール・設計の作法」のことよ。

ブラウザ（JavaScript）がクライアント、Spring Bootがサーバーとなって、ネットワーク越しにデータをやり取りする際に、このルールに従うことでシンプルで統一感のあるシステムになるわ。

### REST APIの4つの原則（基本ルール）
1. **リソースの特定（URI）**
   * 操作する対象（リソース）をURLで表現するの。
   * 例えば、タスクを表すリソースは `/api/tasks` のように**複数形の名詞**にするのが定石よ。`/getTasks` や `/deleteTask` のように動詞をURLに含めるのはRESTの作法に反するわね。
2. **HTTPメソッドによる操作の表現**
   * リソースに対する「操作」は、URLではなくHTTPメソッド（動詞）で表現するわ。
     * **GET**: リソースの取得（タスク一覧を見る、詳細を見る）
     * **POST**: リソースの新規作成（新しいタスクを登録する）
     * **PUT**: リソースの更新（タスクの内容を上書きする）
     * **PATCH**: リソースの一部更新（タスクを「完了」状態にだけ変更する）
     * **DELETE**: リソースの削除（タスクを消去する）
3. **ステートレス（状態を持たない）**
   * サーバー側で「前回の通信状態」を覚えないようにするの。リクエストに必要な情報は、毎回すべて送信するのがルールよ。
4. **データ表現の分離**
   * HTMLそのものを返すのではなく、純粋なデータ（主にJSON形式）だけをやり取りするわ。

## 2. 「1ボタン1API」の設計思想

### 基本的な考え方
お主が言っていた「1つのボタンに対して1つのAPIを用意する」というのは、基本的にとても良い設計思想よ。
これは設計における**「単一責任の原則（Single Responsibility Principle）」**に繋がっているわ。

#### メリット
* **シンプルで頑健**: 1つのAPIがやるべき仕事が1つに絞られるため、コードが読みやすく、バグが発生しにくい。
* **テストが容易**: 「タスクを完了する」だけのAPIなら、その成否のテストも非常にシンプルに書けるわね。
* **再利用性が高い**: モバイルアプリを作る、あるいは他の画面で同じ処理が必要になった際にも、そのAPIをそのまま使い回せるの。

### 注意点と例外（対局のバランス）
何でもかんでも1ボタン＝1APIにしてしまうと、画面の表示や初期化の際、以下のように非効率になることがあるわ。

* **例**: タスク管理画面を開いたとき
  1. タスク一覧を取得するAPIを叩く
  2. 担当ユーザー一覧を取得するAPIを叩く
  3. カテゴリタグ一覧を取得するAPIを叩く
  4. 優先度の設定値一覧を取得するAPIを叩く

画面を開くたびに4回も通信を行うのは、通信のオーバーヘッド（遅延）が発生してパフォーマンスが落ちてしまうわね。
そのため、**「画面表示に必要なデータを一度にまとめて返すAPI（例: `/api/screens/task-dashboard`）」**を1つ用意し、その後の「追加」「編集」「削除」などの個別アクションに対して「1ボタン1API」を割り当てるのが、現実的でスマートな設計よ。

## 3. OpenAPI（Swagger）とは？

「REST API」が設計の概念やルールであるのに対して、**「OpenAPI」**と**「Swagger」**はそれを具体的に活用するための道具よ。

* **OpenAPI**
  * REST APIの仕様（URL、メソッド、送受信するデータの形）をYAMLやJSONで記述するための**「世界標準のフォーマット（記述ルール）」**。
  * 私たちが作成した `openapi.yaml` がこれにあたるわ。
* **Swagger**
  * OpenAPI規格で書かれたファイルを読み込んで、綺麗に画面表示してくれたり（Swagger UI）、ソースコードの雛形を自動でジェネレートしてくれたり（Swagger Codegen）する**「関連ツール群のブランド名」**。

### なぜOpenAPI（APIファースト）で設計するの？
先にAPIの設計図（`openapi.yaml`）をFixさせておくことで、以下の素晴らしい効果が得られるわ。

1. **仕様のズレを防ぐ**
   * フロントエンド（HTML/JS）とバックエンド（Java）を別々に作り始めても、共通の設計図があるから繋ぎ込みの段階で破綻しない。
2. **モックサーバーによる並行開発**
   * 設計図から「偽物のAPIサーバー（モック）」をすぐに立ち上げられるため、Javaの実装が終わっていなくても、フロントエンドの開発を進められるわ。
3. **コードの自動生成**
   * 設計図からJavaのコントローラー（Controller）やデータ型（DTO）のクラスをツールで自動作成できるため、手書きする手間やタイポなどのミスを極限まで減らせるの。

## 4. Thymeleaf（画面）とREST API（Java）の連携イメージ

お主が選んだアプローチ②（APIを使った非同期通信）では、画面の動きは以下のようになるわ。

```
[ブラウザの画面 (Thymeleaf)]
  │
  ├─ 1. ユーザーが「タスク追加」ボタンを押す
  ├─ 2. JavaScriptが入力値を取得し、JSONデータにする { "title": "牛乳を買う" }
  ├─ 3. fetch API でサーバーにPOSTリクエストを送信
  │       │
  │       ▼ (ネットワーク通信 /POST /api/tasks)
  │       │
  ├─ 4. Java側でDBに保存し、完了したタスクのデータをJSONで返却
  │       │
  │       ▲ (ネットワーク通信 / 201 Created + JSON)
  │       │
  └─ 5. JavaScriptが返ってきたJSONを読み取り、HTML（タスクリスト）に新しい行を動的に追加する
```

### JavaScript側のコードイメージ
ブラウザ側（HTML内）で動くJavaScriptは、例えばこんな風にAPIを叩くことになるわ。

```javascript
// タスクを新規登録する関数
function addTask(title) {
    fetch('/api/tasks', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json'
        },
        body: JSON.stringify({ title: title }) // JSONデータに変換
    })
    .then(response => response.json())
    .then(newTask => {
        // 成功したら、返ってきたタスクデータを画面の一覧に動的に追加する
        appendTaskToList(newTask);
    })
    .catch(error => console.error('エラーが発生したわね:', error));
}
```

この連携をこれからJavaの実装とHTMLの作成で実現していきましょう。
まずはゆっくり、一歩ずつ進めば大丈夫よ。
